Loopcraft：人工智能开发中的循环堆叠艺术

⭐ 8.0/10

该文章介绍了由 Peter Steinberger、Boris Cherny 和 Andrej Karpathy 等专家提出的“Loopcraft”概念，即堆叠循环进行人工智能开发，倡导设计循环来提示 AI 代理，而不是使用单次提示。 这种方法可能显著提高 AI 代理的效率和自主性，将范式从手动提示转变为基于循环的自动化系统，影响开发者构建和部署 AI 应用的方式。 Anthropic 公司 Claude Code 负责人 Boris Cherny 表示他不再直接提示 Claude，而是编写循环，这种做法被称为“harness engineering”。Peter Steinberger 强调设计能提示代理的循环，该模式由 Google 工程师 Addy Osmani 命名。

Latent Space · 6月12日 05:34

背景

循环工程和堆叠循环指的是迭代模式，其中 AI 系统在循环中被反复提示以优化输出，从而实现更自主的行为。这与单次提示形成对比，后者每个请求都是独立的。该概念借鉴了 CI/CD 等软件工程实践中的反馈循环，并由 Anthropic 和 Google 的领导者推广。

提出基于 Rust/WASM 的 LLM 边缘语义缓存方案

⭐ 8.0/10

一位开发者提出了一种基于 Rust 和 WebAssembly 构建的开源边缘语义缓存方案，旨在运行于 Cloudflare Workers 等 CDN 边缘节点上。该缓存拦截提示词，使用轻量模型生成嵌入向量，在边缘向量数据库中检查相似度，命中时可在约 5 毫秒内返回缓存响应，避免调用 LLM API。 该方法可显著降低生产环境中 LLM 工作负载的延迟和 API 成本，尤其适用于客服或 RAG 系统中的重复查询。通过将缓存移至边缘，它消除了跨区域网络开销，并实现了无垃圾回收暂停的亚毫秒级代理执行。 该系统使用边缘原生嵌入模型（如 bge-small-en-v1.5）生成向量，采用余弦相似度且阈值设为 0.88 判定缓存命中，并将响应存储在边缘 KV 存储中。Rust 和 WASM 保证了零依赖执行和极小的内存占用，适应边缘运行时限制。

r/MachineLearning · /u/Real-Huckleberry-934 · 6月12日 09:53

背景

LLM 语义缓存基于含义而非精确匹配来存储响应，利用向量嵌入和相似性搜索从缓存中提供相似查询，从而降低成本与延迟。现有方案如 GPTCache 基于 Python 且集中部署，引入了不适合实时边缘环境开销。WebAssembly（WASM）使编译后的代码（如 Rust 代码）能在 CDN 边缘贴近用户运行，具有接近原生的性能且无垃圾回收。

Headroom：压缩 LLM 输入，节省 60-95%的 Token

⭐ 8.0/10

Headroom 是一个新的开源 Python 工具，它能够将工具输出、日志、文件以及 RAG 分块压缩 60-95%，然后再发送给 LLM，同时保持答案质量。 这能显著降低 LLM API 的成本和延迟，使 LLM 应用更高效、更易用，尤其适用于需要大量上下文的场景，如 RAG 和智能体工作流。 Headroom 提供三种模式：Python 库、代理服务和 MCP 服务器。它采用多层处理流水线，包括 CacheAligner 和 ContentRouter，实现无损结构性压缩。

chopratejas · 6月13日 00:00

背景

LLM 处理的是文本单元（Token）。日志或文档等输入可能包含大量 Token，导致成本高昂、响应缓慢。RAG（检索增强生成）系统会将文档拆分为分块以便检索。MCP（模型上下文协议）服务器为 LLM 提供上下文。Headroom 能够高效地压缩这些输入。
